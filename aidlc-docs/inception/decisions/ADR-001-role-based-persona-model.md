## Context and Problem Statement

차량 내 다중 화자 페르소나 정의를 두고 두 가지 모델 중 선택이 필요했다.

초기 설계는 **고정 가족 역할 페르소나**(아빠 / 엄마 / 아이 / 할머니 / 게스트 5종)였으나, 사용자 검토 중 다음 한계가 드러났다.

- 가족 구성 형태가 한 가지로 고정되는 가정 — 실제로는 엄마가 운전자일 수도, 성인 자녀가 운전자일 수도 있음
- 조부모만이 아니라 손주가 있을 수도 있음 (역방향 세대 구성)
- 새로운 가족 구성을 추가하려면 페르소나 프리셋을 추가해야 하고, 그 때마다 CEDAR 정책도 같이 늘어남

본 결정은 페르소나 모델을 어떻게 정의할지 — 어떤 입력 차원(가족 역할 vs 역할 속성)을 기준으로 분기를 만들지 결정한다.

## Considered Options

1. **고정 가족 페르소나 5종** — 아빠 / 엄마 / 아이 / 할머니 / 게스트
2. **역할 속성 기반 페르소나 엔진 + 5종 데모 프리셋** — `driver / age_group / account_owner / relationship` 4 속성 조합으로 페르소나 결정. 데모용으로 5개 프리셋 제공
3. **완전 자유 페르소나 (사용자가 모든 속성 직접 정의)** — 자유도 최대, 데모 시나리오 구성 어려움

## Decision Outcome

**옵션 2 (역할 속성 기반 엔진 + 5 프리셋)을 채택**한다.

핵심 모델:

| 속성 | 타입 | 값 |
|---|---|---|
| `driver` | boolean | true / false (운전석 채널 자동 부여, 트립마다 변경) |
| `age_group` | enum | adult / teen / child / elder |
| `account_owner` | boolean | true / false (차량 계정 소유자) |
| `relationship` | enum | owner / family / guest |

권한과 응답 스타일은 이 4 속성 조합으로 결정되며, CEDAR 정책도 속성 기반으로 작성된다 (FR-02.5).

데모 시연을 위해 5개 프리셋(운전자 성인 / 동승자 성인 / 어린이 / 어르신 / 게스트)을 제공하되, 내부 엔진은 속성 조합으로 동작한다.

## Consequences

- ✅ **유연한 가족 구성**: 누구든 운전석에 앉으면 `driver=true` 자동 부여 — 엄마가 운전해도, 성인 자녀가 운전해도 자연스럽게 운전자 권한 전환 (FR-02.3)
- ✅ **CEDAR 정책 단순화**: 속성 기반 규칙 → 새 가족 조합 추가 시 프리셋 추가 없이 커버 (NFR-04.1)
- ✅ **확장성**: teen 등 새 연령대 추가 시 `age_group` 값만 추가
- ✅ **데모 단순성 유지**: 5 프리셋으로 시나리오 구성, 내부는 역할 엔진 — 데모 슬라이드와 구현이 동시에 깔끔
- ✅ **차별화 어필**: "누구든 운전석에 앉으면 운전자 권한"은 피치 카드로 직접 사용 가능
- ⚠️ **초기 구현 복잡도 약간 증가**: 단순 if-elif 보다 속성 룩업 + CEDAR 표현이 필요. 7시간 타임박스에선 5분 추가 비용 수준
- ⚠️ **데모 시 설명 필요**: "5종이지만 내부는 속성 조합" 한 줄 설명을 피치에 포함

## Prior Art

본 결정은 prior art 없는 **신규 차별화 결정**이다.

비교 대상으로 검토한 LG U+ ixi-drive (커넥티드카 AI 음성 에이전트 프로덕션 시스템)의 메모리 설계(`SPEC-IXICORE-MEMORY-001` v1.2)는 `actor_id = User UUID` 즉 차량 1대당 단일 사용자(주 운전자)를 가정한다. 다중 화자 페르소나 분기 layer는 부재.

본 프로젝트는 이 layer를 추가함으로써 ixi-drive 류 시스템이 커버하지 못하는 가족 / 카풀 / 렌터카 / 택시 시나리오로 확장한다. 이것이 본 프로젝트의 핵심 차별화 포인트다.
